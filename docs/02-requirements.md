# DevFlow 企业研发协作平台需求分析文档

## 1. 引言

### 1.1 编写目的

本文档用于定义 DevFlow 企业研发协作平台的产品需求，明确系统需要解决的问题、目标用户、功能范围、核心业务流程、非功能要求、系统边界及 MVP 版本规划，为后续系统设计、开发、测试和验收提供统一依据。

### 1.2 项目背景

软件研发团队在项目推进过程中，需要持续处理项目成员协作、任务分配、Bug 跟踪、开发进度更新和代码评审等工作。如果缺少统一的协作平台，任务信息容易分散，成员职责与权限边界不清晰，项目进度也难以被及时跟踪。

DevFlow 面向软件研发团队，参考 Jira 的项目管理和 Issue 管理能力，以及 GitHub 的部分 Code Review 流程，模拟真实互联网研发团队的软件协作过程。系统通过统一管理项目、成员、Issue、评论、Review 和通知信息，帮助团队建立清晰、连续的研发协作流程。

DevFlow 不实现完整的 Git 仓库管理，其重点是研发任务及团队协作流程。

### 1.3 术语说明

| 术语 | 说明 |
| --- | --- |
| Issue | 项目中的工作事项，可用于表示 Bug、Feature 或 Task |
| Owner | 项目负责人，负责项目成员与项目任务管理 |
| Developer | 项目开发人员，负责创建、处理 Issue 及提交 Review 请求 |
| Viewer | 项目查看成员，仅查看项目信息和任务状态 |
| Reviewer | 被指定执行 Review 审核的项目成员 |
| Code Review | 对开发处理结果进行审核的简化协作流程，不涉及真实代码仓库管理 |
| MVP | 能够形成核心业务闭环的最小可行产品版本 |

## 2. 产品概述

### 2.1 产品定位

DevFlow 是一个面向软件研发团队设计的企业级研发协作平台，用于管理项目、跟踪任务、管理 Bug 并支持团队成员围绕研发事项开展协作。

系统以项目为协作空间，以 Issue 为核心业务对象，通过成员角色、任务状态、评论和简化 Code Review 串联研发活动。其典型业务过程如下：

```text
用户
  ↓
创建项目
  ↓
添加项目成员
  ↓
创建 Issue / 任务
  ↓
分配开发人员
  ↓
开发处理
  ↓
提交 Review
  ↓
任务完成
  ↓
通知相关成员
```

### 2.2 产品目标

- 为研发团队提供统一的项目与成员管理入口；
- 通过 Issue 集中管理 Bug、Feature 和 Task；
- 明确任务负责人，并持续跟踪任务处理状态；
- 支持团队成员围绕 Issue 进行评论协作；
- 通过简化 Code Review 模拟开发成果审核流程；
- 通过通知向相关成员传递关键协作事件；
- 建立系统角色与项目角色相结合的双层权限模型。

### 2.3 预期价值

- 减少项目、任务和沟通信息分散的问题；
- 提升任务责任归属与状态变化的清晰度；
- 规范从任务创建、开发处理到 Review 完成的协作过程；
- 使不同角色在明确的权限范围内参与项目协作。

## 3. 用户角色分析

### 3.1 权限模型概述

系统采用“系统角色 + 项目角色”的双层权限模型：

- 系统角色用于确定用户在整个平台范围内的职责和权限；
- 项目角色用于确定用户加入具体项目后可执行的操作；
- 同一系统用户可以参与多个项目，并在不同项目中拥有相应的项目角色；
- 项目角色初始固定为 Owner、Developer 和 Viewer，同时在设计上保留未来扩展能力。

### 3.2 系统角色

| 系统角色 | 角色定位 | 主要职责 |
| --- | --- | --- |
| Admin | 系统管理员 | 用户管理、系统权限管理、系统维护 |
| User | 普通系统用户 | 使用平台功能、参与项目协作 |

### 3.3 项目角色

| 项目角色 | 角色定位 | 主要权限 |
| --- | --- | --- |
| Owner | 项目负责人 | 创建项目、管理项目成员、管理项目任务 |
| Developer | 开发人员 | 查看项目、创建和处理 Issue、提交 Review 请求 |
| Viewer | 普通查看成员 | 查看项目信息、查看任务状态 |

项目创建成功后，创建者自动成为该项目的 Owner。其他用户加入项目后，由 Owner 为其分配项目角色，系统依据该角色控制其在项目内可访问的资源和可执行的操作。

## 4. 功能需求分析

### 4.1 用户认证模块

#### 4.1.1 用户注册

- 系统应允许未注册用户通过用户名、邮箱和密码创建系统账号；
- 系统应校验注册信息的完整性；
- 系统应保证邮箱在平台内唯一；
- 注册信息校验通过后，系统应创建用户账号。

#### 4.1.2 用户登录

- 系统应允许用户通过账号和密码登录；
- 系统应校验用户提交的登录信息；
- 登录成功后，用户可以在权限范围内访问系统资源；
- 登录失败时，系统应向用户反馈登录未成功。

#### 4.1.3 用户信息管理

- 用户应能够查看自己的基本信息；
- 用户应能够查看自己参与的项目；
- 系统展示的项目范围应与用户的项目成员关系一致。

### 4.2 项目管理模块

#### 4.2.1 创建项目

- 系统应允许具备创建权限的用户创建项目；
- 创建项目时应填写项目名称、项目描述和项目状态；
- 系统应记录项目创建时间；
- 项目创建成功后，创建者应自动成为该项目的 Owner。

#### 4.2.2 查看项目

- 用户应能够查看自己参与的项目列表；
- 用户应能够进入有权访问的项目并查看项目信息；
- 项目信息应包括项目名称、项目描述、项目状态和创建时间；
- 用户不能查看未参与且无权访问的项目。

#### 4.2.3 项目成员管理

- Owner 应能够向项目中添加成员；
- Owner 应能够从项目中移除成员；
- Owner 应能够修改项目成员的项目角色；
- 成员加入项目后，应获得所分配项目角色对应的权限；
- 成员被移除后，不应继续以该项目成员身份访问项目资源。

### 4.3 Issue 任务管理模块

#### 4.3.1 Issue 数据范围

Issue 用于管理项目中的 Bug、Feature 和 Task。每个 Issue 应包含以下信息：

| 字段 | 需求说明 |
| --- | --- |
| 标题 | 用于概括 Issue 内容 |
| 描述 | 用于说明任务背景和具体内容 |
| 类型 | Bug、Feature 或 Task |
| 优先级 | 用于表示 Issue 的处理优先程度 |
| 创建人 | 创建该 Issue 的用户 |
| 负责人 | 当前负责处理该 Issue 的成员 |
| 所属项目 | Issue 所归属的项目 |
| 创建时间 | Issue 创建时的时间记录 |

#### 4.3.2 Issue 操作需求

- 具备相应项目权限的成员应能够创建 Issue；
- 项目成员应能够查看其权限范围内的 Issue；
- 具备相应权限的成员应能够修改 Issue 信息；
- 系统应提供 Issue 删除能力，并在删除前校验操作权限；
- 具备相应权限的成员应能够为 Issue 分配负责人；
- 负责人应为该项目成员；
- 具备相应权限的成员应能够更新 Issue 状态；
- 系统应记录 Issue 的创建人、所属项目和创建时间。

#### 4.3.3 Issue 状态流转

Issue 状态应按照以下顺序反映任务处理阶段：

```text
Open → In Progress → Review → Done
```

| 状态 | 业务含义 |
| --- | --- |
| Open | Issue 已创建，等待处理 |
| In Progress | Issue 已进入开发处理阶段 |
| Review | 开发处理完成，等待审核 |
| Done | Issue 已完成 |

### 4.4 评论协作模块

- 项目成员应能够在有权访问的 Issue 下添加评论；
- 项目成员应能够查看 Issue 下的评论；
- 评论作者应能够删除自己的评论；
- 系统应限制用户删除其他成员发表的评论；
- 评论内容应归属于对应的 Issue。

### 4.5 Code Review 模块

Code Review 属于增强功能，用于模拟 GitHub Pull Request 的部分审核流程，不涉及真实代码仓库或代码内容管理。

- Developer 应能够针对已处理的 Issue 创建 Review 请求；
- 创建 Review 请求时应能够指定 Reviewer；
- 被指定的 Reviewer 应能够审核 Review；
- 系统应维护 Review 的当前状态；
- Review 状态应包括 Pending、Approved 和 Rejected。

| 状态 | 业务含义 |
| --- | --- |
| Pending | Review 请求已提交，等待 Reviewer 审核 |
| Approved | Reviewer 已通过审核 |
| Rejected | Reviewer 未通过审核 |

### 4.6 通知模块

通知模块属于增强功能，用于提醒用户关注与其相关的协作事件。

- Issue 被分配时，系统应通知对应负责人；
- Review 请求创建后，系统应通知被指定的 Reviewer；
- Owner 邀请用户加入项目时，系统应向被邀请用户发送项目邀请通知；
- 通知内容应与触发通知的业务事件相对应。

## 5. 用户故事

### 5.1 用户注册

作为未注册用户，  
我希望使用用户名、邮箱和密码注册账号，  
以便进入平台并参与研发协作。

### 5.2 用户登录

作为已注册用户，  
我希望通过账号和密码登录系统，  
以便访问权限范围内的项目和任务资源。

### 5.3 创建项目

作为需要发起项目的用户，  
我希望创建项目并填写项目基本信息，  
以便建立团队协作空间并成为项目 Owner。

### 5.4 查看项目

作为项目成员，  
我希望查看自己参与的项目及其基本信息，  
以便了解项目内容和当前状态。

### 5.5 管理项目成员

作为项目 Owner，  
我希望添加或移除项目成员并调整成员角色，  
以便组织项目团队并控制成员权限。

### 5.6 创建 Issue

作为项目 Developer，  
我希望创建 Bug、Feature 或 Task 类型的 Issue，  
以便记录并跟踪项目中的研发事项。

### 5.7 更新 Issue 状态

作为负责处理 Issue 的项目成员，  
我希望更新 Issue 的处理状态，  
以便团队了解任务当前所处阶段。

### 5.8 分配 Issue

作为项目 Owner，  
我希望将 Issue 分配给项目成员，  
以便明确任务负责人并推进任务处理。

### 5.9 评论 Issue

作为项目成员，  
我希望在 Issue 下添加和查看评论，  
以便围绕任务进行沟通并补充相关信息。

### 5.10 提交 Review 请求

作为项目 Developer，  
我希望在完成开发处理后提交 Review 请求并指定 Reviewer，  
以便请求相关成员审核处理结果。

### 5.11 审核 Review

作为被指定的 Reviewer，  
我希望审核 Review 请求并给出通过或拒绝结果，  
以便完成开发处理结果的审核流程。

### 5.12 接收通知

作为平台用户，  
我希望在收到 Issue 分配、Review 请求或项目邀请时获得通知，  
以便及时处理与我相关的协作事项。

## 6. 核心业务流程

### 6.1 创建项目流程

```text
用户登录
  ↓
进入项目管理
  ↓
填写项目信息
  ↓
创建项目
  ↓
创建者成为 Owner
```

流程说明：

1. 用户完成登录并进入项目管理功能；
2. 用户填写项目名称、项目描述和项目状态；
3. 系统校验项目信息并创建项目；
4. 项目创建成功后，系统将创建者设为该项目的 Owner。

### 6.2 创建 Issue 流程

```text
进入项目
  ↓
创建 Issue
  ↓
填写任务信息
  ↓
提交
  ↓
保存 Issue
```

流程说明：

1. 项目成员进入有权访问的项目；
2. 具备创建权限的成员发起 Issue 创建操作；
3. 用户填写标题、描述、类型、优先级、负责人等任务信息；
4. 用户提交 Issue，系统校验信息及操作权限；
5. 校验通过后，系统保存 Issue 并将其关联至当前项目。

### 6.3 Issue 状态流转流程

```text
Open
  ↓
In Progress
  ↓
Review
  ↓
Done
```

流程说明：

1. Issue 创建后处于 Open 状态；
2. 开发人员开始处理任务后，将状态更新为 In Progress；
3. 开发处理完成并进入审核阶段后，将状态更新为 Review；
4. 审核及任务处理完成后，将状态更新为 Done。

### 6.4 项目成员加入流程

```text
Owner 邀请成员
  ↓
成员加入项目
  ↓
分配项目角色
  ↓
获得对应权限
```

流程说明：

1. Owner 发起项目成员邀请；
2. 被邀请用户加入项目；
3. Owner 为新成员分配 Owner、Developer 或 Viewer 项目角色；
4. 系统依据项目角色授予该成员对应的项目访问和操作权限。

## 7. 非功能需求

### 7.1 性能需求

- 系统接口应保持稳定响应，避免因单个请求异常影响其他正常请求；
- 系统应支持一定规模的并发访问，满足团队成员同时进行项目和 Issue 操作的使用场景；
- 项目列表、Issue 列表等常用数据的查询应保持合理的响应效率；
- 通知和增强功能的处理不应明显阻塞核心业务操作。

### 7.2 安全需求

- 系统应在用户访问受保护资源前验证其身份；
- 系统应依据系统角色和项目角色执行权限控制；
- 用户只能访问其权限范围内的项目及相关业务数据；
- 系统应保护用户账号、身份信息和业务数据，防止未授权访问；
- 对创建、修改、删除、分配和审核等操作，系统应在执行前校验权限及数据归属关系。

### 7.3 可维护性需求

- 系统业务模块应职责清晰，便于独立维护；
- 需求和业务边界应保持明确，便于后续扩展项目角色及增强功能；
- 核心业务规则应具备可测试性，支持对认证、权限、项目、Issue 和评论等功能进行验证；
- 功能变更应尽量减少对其他业务模块的影响。

### 7.4 部署需求

- 系统应支持后续容器化部署；
- 部署方式应能够满足应用及其运行依赖的统一管理需求；
- 部署过程应具备可重复性，便于在不同环境中完成系统运行配置。

## 8. 系统边界

### 8.1 系统包含范围

| 功能范围 | 边界说明 |
| --- | --- |
| 用户管理 | 用户注册、登录及用户信息查看 |
| 项目管理 | 项目创建、查看及项目成员管理 |
| Issue 管理 | Bug、Feature、Task 的创建、查看、修改、删除、分配及状态更新 |
| 评论系统 | Issue 评论的添加、查看及本人评论删除 |
| 权限管理 | 系统角色与项目角色双层权限控制 |
| 简化 Code Review | Review 请求创建、Reviewer 指定、审核及状态管理 |
| 通知功能 | Issue 分配、Review 请求和项目邀请通知 |

### 8.2 系统不包含范围

- 完整 Git 仓库管理；
- 在线代码编辑器；
- CI/CD 流水线；
- 微服务架构；
- 复杂前端系统。

## 9. MVP版本规划

### 9.1 Version 1.0（核心业务版本）

Version 1.0 以完成研发协作的核心业务闭环为目标。

| 功能模块 | 版本范围 |
| --- | --- |
| 用户认证 | 用户注册、用户登录、用户信息查看 |
| 项目管理 | 项目创建、项目查看、项目基本信息管理 |
| 项目成员管理 | 添加成员、移除成员、修改项目角色 |
| Issue 管理 | Issue 创建、查看、修改、删除、分配及状态更新 |
| 评论系统 | 评论添加、查看及本人评论删除 |
| 基础权限控制 | 系统角色与项目角色的基础访问控制 |

版本完成标准：用户能够登录平台，创建项目并组织项目成员；项目成员能够创建、分配、处理和完成 Issue，并通过评论开展任务协作，从而形成完整的核心业务闭环。

### 9.2 Version 1.1（工程增强版本）

Version 1.1 在核心业务稳定运行的基础上增加工程增强能力，提升系统的协作体验和项目技术亮点。

| 增强项 | 版本范围 |
| --- | --- |
| Redis 缓存 | 为系统增加缓存能力 |
| Celery 异步任务 | 支持后台异步任务处理 |
| 通知系统 | 支持 Issue 分配、Review 请求和项目邀请通知 |
| Code Review | 支持 Review 请求、Reviewer 审核和 Review 状态管理 |
| WebSocket 实时通知 | 提升通知到达的实时性 |

版本完成标准：在不改变核心业务定位的前提下，完成通知与简化 Code Review 流程，并增强系统在缓存、异步处理和实时消息方面的工程能力。
