# DevFlow 企业研发协作平台

## 1. 项目简介

DevFlow（DevFlow Platform）是一个面向软件研发团队设计的企业级研发协作平台，用于模拟真实互联网企业中的项目管理、任务协作、代码评审与消息通知流程。

平台以项目和 Issue 为核心，支持项目负责人组织成员与推进任务，支持开发人员创建和处理 Issue、提交 Review 请求，并通过角色权限控制和通知机制保障协作流程清晰、有序。DevFlow 参考 Jira 的项目管理与 Issue 管理能力，以及 GitHub 的部分代码协作流程，但不以实现完整的 Git 托管平台为目标。

项目面向个人 GitHub 作品展示及 Python 后端、AI 应用开发方向的秋招面试场景，在合理的个人开发规模内，重点呈现企业级后端架构、权限系统、缓存优化、异步任务和容器化部署等工程实践。

## 2. 项目背景

在真实的软件研发团队中，项目推进通常涉及成员管理、任务拆分、负责人分配、开发处理、代码评审和消息通知等多个环节。不同角色需要在统一的平台中协同工作，并依据职责访问相应的系统资源。

DevFlow 通过还原上述典型协作场景，为企业级 Python 后端开发提供一套完整、可实践的项目载体。项目主要用于学习和实践以下方向：

- Python 后端工程开发；
- FastAPI 企业级项目架构设计；
- 关系型数据库设计；
- 身份认证与权限系统设计；
- Redis 缓存设计；
- 异步任务处理；
- Docker 容器化部署。

## 3. 项目目标

DevFlow 的核心目标是构建一套职责明确、流程完整、架构清晰的研发协作平台，模拟软件研发团队从项目创建到任务完成的主要工作过程：

```text
用户
  ↓
创建项目
  ↓
添加项目成员
  ↓
创建任务 / Issue
  ↓
分配开发人员
  ↓
开发处理
  ↓
代码 Review
  ↓
任务完成
  ↓
通知相关成员
```

项目具体目标包括：

- 建立统一的用户认证和角色权限控制机制；
- 支持项目、成员和项目状态的规范化管理；
- 围绕 Issue 实现任务创建、分配、流转和完成的完整闭环；
- 模拟 GitHub Pull Request 场景下的 Code Review 流程；
- 通过通知机制同步 Issue 分配、Review 请求和系统消息；
- 使用缓存、异步任务和容器化部署提升系统的工程完整性。

## 4. 用户角色

系统采用基于角色的访问控制，根据用户职责划分为以下三类角色：

| 角色 | 定位 | 主要职责 |
| --- | --- | --- |
| Admin | 系统管理员 | 负责用户管理、权限管理和系统维护 |
| Project Manager | 项目负责人 | 负责创建项目、管理项目成员、分配任务和管理项目进度 |
| Developer | 普通开发人员 | 负责查看参与项目、创建 Issue、处理任务和提交 Review 请求 |

不同角色仅能访问其职责范围内的功能和资源，从而保证系统操作边界清晰、项目协作过程可控。

## 5. 核心功能

### 5.1 用户认证与权限模块

用户认证与权限模块负责识别用户身份并控制资源访问，是平台安全机制的基础。

- 用户注册；
- 用户登录；
- 基于 JWT 的身份认证；
- 基于 RBAC 的角色权限控制。

### 5.2 项目管理模块

项目管理模块用于维护项目基本信息、成员关系及项目状态，为后续任务协作提供统一的项目上下文。

- 创建项目；
- 项目信息管理；
- 项目成员管理；
- 项目状态管理。

### 5.3 Issue 任务管理模块

Issue 任务管理是 DevFlow 的核心模块，覆盖任务从创建、分配到评审完成的主要生命周期。

- 创建和修改 Issue；
- 分配 Issue 负责人；
- 设置 Issue 优先级；
- 管理 Issue 状态。

Issue 支持以下类型：

| 类型 | 说明 |
| --- | --- |
| Bug | 用于记录和跟踪缺陷修复任务 |
| Feature | 用于记录和跟踪功能开发需求 |
| Task | 用于记录和跟踪一般研发任务 |

Issue 状态按照以下流程流转：

```text
Open → In Progress → Review → Done
```

### 5.4 评论与协作模块

评论与协作模块用于围绕 Issue 进行信息补充和团队沟通，保证任务背景、处理过程及讨论内容集中留存。

- Issue 评论；
- 评论回复。

### 5.5 Code Review 模块

Code Review 模块模拟 GitHub Pull Request 的评审流程，用于衔接开发处理与任务完成阶段。

- 开发人员提交 Review 请求；
- Reviewer 执行审核；
- Review 状态管理。

Review 支持以下状态：

| 状态 | 说明 |
| --- | --- |
| Pending | Review 请求已提交，等待审核 |
| Approved | Review 已通过 |
| Rejected | Review 未通过 |

### 5.6 通知系统

通知系统负责向相关成员同步关键协作事件，减少任务流转过程中的信息遗漏。

- Issue 分配通知；
- Review 请求通知；
- 系统消息提醒。

## 7. 项目亮点

### 7.1 企业级后端架构设计

项目采用模块化、分层化的后端架构，按照业务职责组织代码，明确接口、业务逻辑和数据访问等层次的边界。清晰的代码结构有助于降低模块耦合度，并提升项目的可维护性和可扩展性。

### 7.2 RBAC 权限系统

系统围绕 Admin、Project Manager 和 Developer 三类角色设计 RBAC 权限控制机制，使不同角色只能访问和操作其职责范围内的资源。该设计能够体现企业应用中身份、角色、权限与业务资源之间的访问控制关系。

### 7.3 Redis 缓存优化

项目使用 Redis 缓存热点数据，减少数据库重复查询，提升高频接口的响应速度。缓存能力与具体业务模块结合，在保证数据访问效率的同时保持系统架构清晰。

### 7.4 异步任务处理

项目采用 Celery 与 Redis 处理消息通知和后台任务，将非实时业务从核心请求流程中解耦，缩短接口同步处理时间，并提升任务处理过程的可维护性。

### 7.5 Docker 容器化部署

项目通过 Docker 管理后端服务、MySQL 数据库和 Redis 运行环境，统一开发与部署所需的基础设施配置，降低环境差异带来的部署成本，并提高项目的可复现性。
